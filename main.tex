\begin{abstract}
This paper reports a reproducible numerical-analysis benchmark for block-based image decryption using the 24-image Kodak lossless benchmark set \cite{kodak-lossless}. Each image is resized to $128 \times 128$, partitioned into $8 \times 8$ blocks, encrypted with dense and sparse key matrices, corrupted with additive Gaussian noise, and then decrypted by Gaussian Elimination with partial pivoting, LU Decomposition, Gauss-Seidel with relaxation, GMRES, and a NumPy linear-solve baseline. The study emphasizes numerical robustness, reconstruction quality, convergence behavior, and runtime under controlled perturbations. The generated artifact bundle includes Kodak manifests, wall-of-decryption figures, convergence traces, delta-versus-baseline summaries, and LaTeX-ready tables for direct inclusion in the manuscript.
\end{abstract}

\section{Introduction}
Matrix-based image encryption can be expressed as a repeated sequence of linear systems. For a vectorized plaintext block $x \in \mathbb{R}^{64}$ and key matrix $K \in \mathbb{R}^{64 \times 64}$,
\[
    c = Kx,
\]
and decryption under transmission noise is modeled as
\[
    c' = c + \epsilon, \qquad K\hat{x} = c'.
\]
The quality of the recovered image therefore depends on the numerical behavior of the solver used to invert the block system, especially when the key structure is sparse or when the encrypted blocks are perturbed by stochastic noise.

This work is a numerical analysis study of matrix-based decryption workflows, not a security proof of modern cryptographic primitives. The goal is to compare robustness, runtime, and reconstruction fidelity across direct, iterative, and hybrid solvers under the same Kodak-only benchmark protocol.

\subsection{Pipeline}
The experimental workflow follows the pseudocode-driven plan used for the notebook update.
\begin{enumerate}
\item Convert each benchmark image to grayscale and resize it to $128 \times 128$.
\item Partition the image into $8 \times 8$ blocks, yielding $64$-dimensional linear systems.
\item Generate a dense key matrix and a sparse key matrix.
\item Encrypt each block and inject additive Gaussian noise before decryption.
\item Recover each block with multiple solvers and compare numerical quality and cost.
\item Generate graphical visualizations from the numerical outputs for analysis and reporting: dataset contact sheets, solver metric graphs, Gauss-Seidel convergence plots, wall-of-decryption image panels, absolute-error heatmaps, and the paper triptych.
\end{enumerate}

The sparse key is intentionally used alongside the dense key because solver behavior can change substantially when the coefficient matrix contains many zeros or near-decoupled interactions.

The graphical-visualization stage is not cosmetic. It is part of the analysis pipeline itself: the contact sheet documents the exact Kodak inputs, the metric graphs summarize solver behavior across noise levels, the convergence plots expose iterative dynamics, and the wall, reconstructed-image, and error-heatmap figures make local reconstruction failures visible in ways that scalar metrics alone cannot.

\section{Methods}
\subsection{Solvers}
The notebook implements the following methods:
\begin{itemize}
\item \textbf{Gaussian Elimination with partial pivoting}: direct method used as the classical baseline for row-wise elimination stability \cite{chapra-canale-2010}.
\item \textbf{LU Decomposition (Doolittle)}: factorization-based method that amortizes the matrix factorization across repeated block solves \cite{chapra-canale-2010,lu-notes}.
\item \textbf{Gauss-Seidel with relaxation}: iterative method used to study convergence and noise sensitivity in repeated fixed-point updates.
\item \textbf{GMRES}: Krylov-subspace hybrid method added to match the target plan and to provide a modern iterative reference.
\item \textbf{NumPy solve}: baseline reference built on dense linear algebra routines, used for delta-versus-baseline analysis.
\end{itemize}

\subsection{From Pseudocode to Code}
The notebook was rewritten so that each solver mirrors the structure of the supplied pseudocode rather than hiding the numerical work inside a library call.

For Gaussian Elimination, the pseudocode specifies partial pivoting with row exchanges, multiplier storage, and back substitution. The notebook implementation follows that pattern directly: it builds an augmented matrix, tracks row order through a pointer array, swaps pivot rows, stores the elimination multipliers below the diagonal, and then performs explicit backward substitution to recover the block solution.

For LU factorization, the pseudocode uses the Doolittle-style in-place decomposition. The notebook follows the same sequence: each column is normalized by the pivot, the Schur-complement update is applied to the trailing submatrix, and the resulting packed matrix is reused by a separate forward/back substitution routine for every encrypted image block.

For Gauss-Seidel, the pseudocode repeatedly updates the current iterate using the newest available values, checks an approximate stopping condition, and records whether the iterate reached the iteration cap. The notebook reflects that structure exactly. It normalizes by the diagonal, applies the relaxation factor during each component update, measures a relative residual after every sweep, and optionally stores the residual history for the convergence-trace plots.

For GMRES, the notebook intentionally delegates the Krylov basis construction to SciPy because the project goal is comparison rather than re-deriving Arnoldi from scratch. Even so, the surrounding experiment logic mirrors the pseudocode plan: the same noisy block systems are passed to the solver, iteration counts are recorded through a callback, and the resulting residuals are exported in the same artifact format as the hand-coded methods.

\subsection{How the Notebook Works}
The code path from raw Kodak image to final table is fully deterministic once the random seed is fixed.
\begin{enumerate}
\item The configuration cell selects the Kodak subset, the block size, the noise levels, and the solver tolerances.
\item The dataset loader converts each image to grayscale, resizes it to $128 \times 128$, stores provenance metadata, and writes a contact sheet for the exact 24 images used in the run.
\item Two key matrices are generated: one dense and one sparse, each with a diagonal boost so that the block systems remain numerically solvable across all repeated experiments.
\item Each image is split into $8 \times 8$ blocks, flattened into $64$-vectors, encrypted by matrix multiplication, then perturbed with Gaussian noise.
\item Every solver is run on the same noisy blocks. The code measures reconstruction fidelity, residuals, runtime, mean iterations, and whether any block solve reached the iteration cap.
\item The export stage aggregates those per-image records into CSV summaries, LaTeX tables, metric figures, convergence plots, and wall-of-decryption panels that the paper includes directly.
\end{enumerate}

\subsection{Independent Variables}
The notebook varies two main factors.
\begin{itemize}
\item \textbf{Matrix type}: dense and sparse key matrices.
\item \textbf{Noise level}: Gaussian noise with $\sigma \in \{0.00, 0.01, 0.03, 0.07, 0.12\}$.
\end{itemize}

Each condition is repeated three times across all 24 Kodak images. The run configuration and machine metadata are stored in \texttt{out/config.json} and \texttt{out/run\_info.json}.

\subsection{Evaluation Metrics}
The paper uses the artifacts generated by the notebook and post-processing scripts.
\begin{itemize}
\item PSNR and MSE for reconstruction quality.
\item Relative error and relative residual for numerical accuracy.
\item Median runtime and interquartile range for efficiency.
\item Success rate for robustness under each condition, defined as a complete solve with no failed blocks.
\item Full-convergence rate, defined more strictly as a solve that also avoids hitting the configured iteration cap.
\item Gauss-Seidel convergence traces for iterative behavior.
\item Delta versus \texttt{np\_solve} for baseline-relative comparison.
\end{itemize}

\subsection{Results Highlights}
Two high-level outcomes stand out in the regenerated summaries.

First, the fastest solver overall is the NumPy baseline. Averaged over all noise levels, its median solve time is about $6.80$ ms for dense keys and $6.82$ ms for sparse keys. Among the custom methods, LU is the speed leader, with average median times of about $244.71$ ms for dense keys and $246.03$ ms for sparse keys, substantially ahead of Gaussian elimination, GMRES, and Gauss-Seidel.

Second, the strongest convergence behavior among the iterative methods is delivered by GMRES. Its full-convergence rate is $1.0$ for both dense and sparse matrices across all tested noise levels, whereas Gauss-Seidel drops to an overall full-convergence rate of $0.833$ because some zero-noise cases hit the iteration ceiling before satisfying the strict stopping rule. At the same time, all methods achieve a solve-success rate of $1.0$, so the practical difference is not reconstruction failure but how reliably each method satisfies the configured convergence criterion.

\subsection{Interpretation}
The quantitative tables and figures are designed to answer four questions.
\begin{enumerate}
\item How much reconstruction quality is lost as Gaussian noise increases?
\item Does sparse structure help or hurt solver behavior under the chosen diagonal boosts?
\item How large is the runtime gap between direct, factorization-based, iterative, and hybrid methods?
\item How far do the custom solver implementations deviate from a dense linear-solve baseline?
\end{enumerate}

The artifact set generated by the notebook is intended to support those questions directly rather than through manually copied statistics.

\section{Discussion}
Three practical observations motivate the current design.

First, LU decomposition is expected to remain highly competitive whenever the same key matrix is reused across many blocks, because the factorization cost is paid once and the subsequent solves are comparatively cheap.

Second, Gauss-Seidel and GMRES provide complementary iterative viewpoints. Gauss-Seidel exposes convergence behavior directly through residual traces, while GMRES offers a hybrid Krylov-based reference for the same noisy block systems. The updated reporting separates \emph{solve success} from \emph{full convergence}. That distinction matters most at zero noise, where Gauss-Seidel still reconstructs every block correctly but some runs terminate at the configured iteration ceiling before satisfying the strict stopping test.

Third, the dense-versus-sparse comparison matters numerically even outside classical conditioning summaries. Two matrices can have similar condition numbers and still differ in residual propagation, convergence behavior, and runtime profile once the solver and noise process are fixed.
